# 显影 · Unearth
### *A Memory Excavation*

> 我们不是在整理文件。我们是在选择带走什么。

---

## 项目灵魂

四五年、三个以上的国家、四五万张照片。这些照片是一个人对自己人生的原始记录。

整理它们，不是清理硬盘的行为，而是一次**记忆的考古**——从时间的地层里，把最重要的那些挖出来，带进往后的人生。

**核心哲学：**
- 没有「删除」，只有「留在这里」——它留在时间里，但不在你的故事里
- 没有「保存」，只有「带走」——主动的选择，而不是被动的保留
- 整理的过程本身，应该是一种享受，而不是负担

---

## 三阶段路线图

### 阶段一：私人工具（当前目标）
- 服务对象：单一用户，在 Mac 本地运行
- 目标：能用、高效、满足个人工作流
- 照片来源：外置硬盘，按年月组织的 Lightroom 导出结构
- 可以有硬编码的个人偏好

### 阶段二：本地 APP（下一步）
- 开始有独立的「人格」和美学
- 引入 AI 生成功能（人生摘要）
- 界面达到「高艺术感」的标准
- 可打包为独立 Mac 应用

### 阶段三：可复用工具（未来）
- 抽象出可配置的部分（硬盘路径、用户名、语言）
- 个人品味变成「可配置的默认值」
- 开源或小范围分发给朋友使用
- 架构从第一天起就需要考虑这一点

## 用户画像（阶段一）

- **用户：** 做文化洞察的创意工作者，有强烈的个人审美
- **设备：** MacBook，外置硬盘通过 USB 挂载
- **照片：** ~40,000 张，~900GB，索尼和富士相机拍摄
- **格式：** ARW（索尼 RAW）、RAF（富士 RAW）、JPEG、HEIF、HEIC、PNG
- **结构：** 按 `年/月/` 文件夹组织，含 Lightroom XMP sidecar 文件
- **目标：** 删除约 1/3 文件，整理剩余照片，最终用于策展或摄影书

## 核心功能规格

### F1：扫描与建立索引

后端启动时扫描指定根目录，建立本地 SQLite 数据库：

**扫描内容：**
- 所有图片文件（ARW、RAF、JPEG、JPG、HEIF、HEIC、PNG）
- 对应的 XMP / XML sidecar 文件（同名，不同后缀）
- 从 EXIF 读取：拍摄时间、GPS 坐标（如有）、相机型号

**分组逻辑：**
1. 按年/月分组（来自文件夹结构 + EXIF 校验）
2. 在月内，按拍摄时间间隔自动聚合「拍摄事件」（同一事件内照片时间间隔 < 30 分钟）
3. 在事件内，识别 RAW/非RAW 配对（同名文件，不同后缀）

**配对规则：**
```
photo_001.ARW  ←→  photo_001.JPG  （索尼 RAW + JPEG 配对）
photo_001.RAF  ←→  photo_001.JPG  （富士 RAW + JPEG 配对）
photo_001.XMP               （sidecar，跟随主文件）
```

---

### F2：地层全景界面（主界面）

**概念：** 用户进入 APP，看到的是一个地质切面——你的 4-5 年以「地层」的形式呈现。

**视觉规则：**
- 每一年是一个地层，从上（最近）到下（最早）排列
- 地层色温：越深（越早）越暖，越近（越新）越冷
  - 2024：冷蓝白 `#E8F0FF`
  - 2023：浅灰蓝 `#C5D8E8`
  - 2022：暖米白 `#E8D8C0`
  - 2021：琥珀暖 `#D4A870`
  - 2020：深琥珀 `#B8864E`
- 每个月是地层里的一个「区块」，显示照片数量
- 已处理的区块：清晰、发光
- 未处理的区块：磨砂、半透明

**交互：**
- 鼠标悬停某个区块：显示「XX年XX月 · N张照片 · N个拍摄事件」
- 点击进入：平滑缩放动画，进入该月的考古现场

---

### F3：考古现场界面（章节视图）

**概念：** 进入某月后，看到该月所有「拍摄事件」的概览，像考古现场的格子分区。

**显示内容：**
- 每个事件显示：封面缩略图、照片数量、大致时间、地点（如有 GPS）
- 事件状态：未开始 / 进行中 / 已完成

**进入事件：** 点击进入单个拍摄事件，开始核心决策流程。

这是用户花时间最多的地方。

**布局：**

│  █                                                    █  │
│  ████████████████████████████████████████████████████  │
│                                                         │
│  [  ○ ● ○ ○ ● ○ ○  ]  ← 事件内照片缩略图横条            │
│                                                         │
│     [← 留在这里]    [↑ 稍后]    [带走 →]               │

**快捷键：**
| 键 | 动作 |
|---|---|
| `→` 或 `K` | 带走（保留） |
| `←` 或 `D` | 留在这里（软删除） |
| `↑` 或 `S` | 稍后决定（跳过） |
| `Space` | 放大查看原图细节 |
| `Z` | 撤销上一步 |
| `Star / F` | 标记为「书候选」（入围摄影书/策展） |

**RAW+JPEG 配对处理：**
当前照片若有配对文件，底部显示小提示：「此照片有配套 RAW 文件」

### F5：删除逻辑（关键）

**「留在这里」触发的操作：**

```
情况A：RAW + JPEG 配对存在
  → 标记 RAW 文件为「待删除」
  → 保留 JPEG 文件
  → 删除对应 XMP sidecar（如有）

情况B：仅有非RAW文件（JPEG / HEIF / HEIC / PNG）
  → 标记该文件为「待删除」
  → 删除对应 XMP sidecar（如有）

情况C：仅有 RAW 文件（无 JPEG 配对）
  → 标记 RAW 文件为「待删除」
  → 删除对应 XMP sidecar（如有）
```

**软删除机制（安全第一）：**
1. 「留在这里」不立即删除，而是移动到 `{根目录}/_unearth_staging/` 文件夹
2. Staging 文件夹每次会话结束后，弹出确认界面：「你本次留下了 N 个文件，共 X GB，确认永久释放？」
3. 确认后，staging 文件夹被清空
4. 用户可以在 staging 文件夹里反悔，重新取出文件

---

### F6：进度与统计

**实时显示：**
- 本次会话：处理了多少张 / 带走多少 / 留在这里多少
- 释放空间：已释放 / 预计总释放
- 总进度：N% 已完成

**「书候选」专区：**
- 单独的视图，显示所有标记了 `★` 的照片
- 可以导出候选清单（JSON / CSV），供后续导入 Lightroom

---

### F7：AI 人生摘要（阶段二，提前预留接口）

所有照片处理完成后，调用 Anthropic Claude API，生成一份「考古报告」：

**输入数据：**
- 每月/每年带走 vs 留下的比例
- GPS 最常出现的城市（前10）
- 相机使用频率（索尼 vs 富士）
- 拍摄时间分布（早晨/白天/傍晚/夜晚）
- 「书候选」照片的时间/地点分布

**输出格式：** 一段 300-500 字的中文叙述性文字，风格如同考古报告，但有诗意。

**示例输出风格：**
> 「在你保留下来的记忆里，英国的冬天占据了最多的篇幅——不是因为那里最美，而是因为你在那里花了最多时间等待光线。你在中国拍了很多人，在欧洲拍了很多窗户。有意思的是，巴黎你只去了四天，却带走了那段旅程里密度最高的照片……」

**API 调用：** 后端 `/api/summary/generate`，前端展示为全屏仪式感动画。

## 视觉语言规范（给 Claude Code 前端）

### 核心美学方向
**「发光的地下室」** — 黑暗中有温度，现代中有质感

### 色彩系统
```css
--color-void: #0A0A0F;           /* 最深背景，接近黑色 */
--color-deep: #12141E;            /* 主背景 */
--color-surface: #1C1F2E;         /* 卡片/面板背景 */
--color-glass: rgba(255,255,255,0.06); /* 玻璃效果 */
--color-glass-border: rgba(255,255,255,0.12);

/* 地层色彩系统 */
--strata-2024: #A8C5E8;
--strata-2023: #8FADC4;
--strata-2022: #C4A882;
--strata-2021: #D4956A;
--strata-2020: #C47840;

/* 动作色彩 */
--color-keep: #7EB8A4;            /* 带走：冷绿色，有生命感 */
--color-leave: #8B7355;           /* 留在这里：土褐色，归于尘土 */
--color-star: #E8C878;            /* 书候选：琥珀金 */
```

### 材质规范
- **玻璃态面板：** `backdrop-filter: blur(20px)` + 低透明度白色背景
- **流体波纹：** 每次决策操作触发 CSS ripple 动画，从操作点向外扩散
- **高斯模糊背景：** 当前照片铺满屏幕背景，`filter: blur(40px) brightness(0.3)`
- **地层纹理：** 地层区块使用细腻的噪点纹理，`filter: url(#noise)`

### 字体方向
- **标题/导航：** 具有设计感的衬线字体或几何无衬线（避免 Inter/SF Pro）
- **正文/数据：** 等宽或轻量级字体，冷静、克制
- **数字：** tabular-nums，保持对齐

### 动画原则
- 所有过渡：`cubic-bezier(0.4, 0, 0.2, 1)`，有阻尼感
- 照片切换：150ms fade，不要滑动（太轻浮）
- 「留在这里」：向下沉入，`scale(0.95) translateY(10px) opacity(0)`
- 「带走」：向上浮起，`scale(1.02) translateY(-5px)` 然后收进角落
- 涟漪扩散：`0.6s ease-out`，从操作点向外圆形扩散

## 非功能性要求

### 性能
- 预览图生成：后台异步，不阻塞 UI；JPEG 预览最大 1200px 长边，≤300KB
- RAW 预览：使用 embedded JPEG preview（不全量解码 RAW），响应时间 <500ms
- 扫描 4 万张照片：应在 2 分钟内完成索引（不生成预览，仅建数据库）

### 安全
- **绝对不直接删除文件**，所有「留在这里」操作先移到 staging 文件夹
- 原始文件所在目录只读操作，写操作只在 staging 文件夹和 data/ 目录
- 每次会话开始前，检查外置硬盘是否挂载

### 断点续传
- 所有决策状态存入 SQLite，关闭 APP 后重新打开可继续
- 进度不丢失

# 显影 v2 产品方向 · 追加内容

---

## V2 核心方向：从「线性整理」到「非线性发现」

### 产品本质的重新定义

V1 是：你来评判照片。
V2 是：**系统帮你读懂自己。**

线性挖掘（按时间一张一张）必然带来疲劳，因为用户永远不知道终点在哪。
非线性发现解决了这个问题——系统先做功课，以有意义的方式呈现，用户带着好奇心进入，而不是责任感。

---

### 两种核心模式

---

#### 模式一：主题模式（你的执念）

**本质：** 系统帮你发现你自己都没意识到的拍摄执念。

你拍了多少扇窗？多少棵树？多少只猫？你不知道，但数据库知道。

**体验：**
系统将跨越不同时间、不同地点的相似照片聚合在一起，呈现为一个「主题胶卷」。用户进入之后看到的不是「2021年11月3日的照片」，而是「你拍过的所有雨天」或「你镜头里反复出现的窗」。

**触发的情绪：** 自我发现。「原来我拍了 847 张窗户。」这个发现本身就是礼物。

**实现路径（分阶段）：**

*现在可以做（基于现有 EXIF 数据）：*
- 地点执念：GPS 聚类，找出你反复回去拍的地方
- 时间执念：同一时间段（如每天傍晚5-7点）的照片聚合
- 设备执念：索尼机身 vs 富士机身的拍摄习惯对比

*之后加（需要 AI 视觉分析）：*
- 内容识别：树、窗、人、猫、食物、建筑、天空
- 情绪分类：暖光 / 冷光 / 孤独感 / 热闹感 / 人群
- 构图偏好：中心构图 / 边缘构图 / 留白习惯

---

#### 模式二：故事模式（跨越时间的叙事）

**本质：** 系统把碎片重新编排成有意义的叙事。

不同日子、不同国家、相似的光线或地理位置——系统说：这几张照片，其实是同一个故事。

**体验：**
每次进入故事模式，系统生成一个「今日故事」——可能是「你在三个不同城市拍过的早餐」，可能是「2020年和2023年，同一个地点的你」，可能是「一整年里，每个月的第一天」。

故事有开头、有中段、有结尾。你在里面做决策，但感受的是叙事，不是评审。

**触发的情绪：** 意义生成。你以为那些照片是孤立的，系统告诉你它们之间有连接。

**实现路径（基于现有数据）：**
- 跨时期同地点：相同 GPS 坐标，不同年份
- 同一天的完整故事：某年某月某日，从早到晚
- 地理漫游：按移动轨迹重现一次旅行
- 对比叙事：英国的冬天 vs 中国的冬天

---

### 每日限额机制

**这是解决「整理疲劳」最重要的设计。**

不是「什么时候才能整理完」，而是「今天的故事已经看完了」。

```
每日进入 APP：
→ 系统为今天准备了一个「故事」或一个「主题」
→ 完成这个单元，今日份额用完
→ 明天系统再准备新的

用户心理：像每日一首诗，不是一次读完整本书。
```

**技术实现：**
- 每日生成逻辑：根据未处理照片池，算法选取最有「叙事张力」的组合
- 份额大小：默认 30-50 张/天（可配置）
- 如果用户想继续：可以解锁「加时」，但有轻微阻力（不是直接按钮，需要一个小确认）

---

### 对现有框架的影响

**保留的：**
- 带走 / 留在这里 / 稍后 三个核心决策——够用，不改
- 软删除 staging 机制——安全性不变
- 书候选（★）标记——保留

**需要新增的后端能力：**
- `GET /api/themes` — 返回可用主题列表（基于 EXIF 聚类）
- `GET /api/themes/{theme_id}/photos` — 某主题下的照片
- `GET /api/story/today` — 今日故事（系统生成）
- `POST /api/story/generate` — 手动触发新故事生成
- 聚类算法：GPS 聚类、时间段聚类（后端 Codex 负责）

**需要新增的前端界面：**
- 主题浏览器（StrataView 的平行入口）
- 故事播放器（类似 DecisionView，但有叙事上下文）
- 今日故事入口（主界面增加一个「今天」的入口）

### 实现优先级

```
现在（V1 收尾）：
  → 完成线性挖掘流程的基础体验
  → 加入「记忆上下文层」（时间/地点/第几天）
  → 完成删除验证和 UI 基础动画

V2 第一步（数据已有，可以做）：
  → 故事模式：跨时期同地点 + 同一天完整故事
  → 每日限额机制
  → 主题模式初级版：GPS 聚类 + 时间段聚类

V2 第二步（需要 AI 视觉）：
  → 接入 Claude API 做照片内容标签
  → 主题模式完整版：内容识别
  → 情绪/色调聚类
```

# 显影 V3 叙事引擎：五部小说

---

## 核心理念

同一批照片，可以是五部完全不同的小说。

不是「视图切换」，是叙事框架的根本转换。每一种框架借鉴一部文学经典的结构基因，让用户以截然不同的方式进入自己的记忆。选择哪部小说阅读，决定了你今天看到的是什么样的自己。

---

## 五部小说

---

### 第一部：《哈扎尔词典》模式
**原著：** 米洛拉德·帕维奇（Milorad Pavić）

**文学基因：** 这本书没有正确的阅读顺序。任何一个词条都是入口，每次进入看到的是不同的叙事网络。同一个事件从三个相互矛盾的视角描述，读者自己拼凑真相。

**照片映射：**
你的照片库是一本词典。每一个 EXIF 属性都是一个词条——「长焦」、「夜晚」、「浅景深」、「富士机身」、「凌晨五点」。点击任何一个词条，进入的是跨越不同年份、不同地点、真正在「对话」的照片群。没有时间轴，没有固定起点。

**技术基础（现有数据即可实现）：**
- 焦距分类：广角（<35mm）/ 标准（35-70mm）/ 长焦（>70mm）
- 曝光分类：日光（ISO<800）/ 低光（ISO 800-3200）/ 夜晚（ISO>3200）
- 景深分类：浅（f/1.2-2.8）/ 中（f/2.8-8）/ 深（f/8+）
- 时刻分类：破晓 / 清晨 / 午间 / 黄金时段 / 傍晚 / 深夜
- 主动性标记：RAW = 深思熟虑的拍摄；仅JPEG = 随手抓拍
- 组合词条：「黄金时段 × 长焦 × 富士」构成一个独特词条，跨越所有年份

**UI 方向：**
一个暗色的「词典入口」——不是网格，是词条云或分类索引。每个词条显示照片数量。点击进入，照片以「对话排列」呈现：左列是2019-2021年，右列是2022-2023年，相似特征的照片彼此呼应。

---

### 第二部：《酒吧长谈》模式
**原著：** 马里奥·巴尔加斯·略萨（Mario Vargas Llosa）

**文学基因：** 两条叙事线在一家酒吧的对话中交织，不断插入不同时期的记忆，倒叙和插叙构成一张时间的网。「秘鲁什么时候开始堕落的？」——一个问题触发无数条记忆线同时涌现。

**照片映射：**
系统在你的照片库里寻找「跨时间的回响」——2019年11月的一个下午和2023年11月的一个下午，同样的光线、同样的焦距、同样的时刻。把它们并置，让时间自己说话。一张照片触发另一条时间线，另一条时间线再触发另一段记忆，直到展开成一张交错的网。

**技术基础：**
- 「回响算法」：匹配 时刻分类 + 焦距分类 + 曝光分类 三个维度相同的照片对
- 权重：三个维度全中 = 强回响；两个维度 = 弱回响
- 构建 `temporal_echoes` 关系表：存储照片对及回响强度
- 叙事线生成：从任意一张照片出发，沿回响链展开，最长链即为「一个故事」

**UI 方向：**
非线性时间轴——不是从左到右，而是从中心向外扩散。一张照片在中心，回响照片环绕在外，年份越远颜色越深。用户可以「跳入」任何一张照片，成为新的中心。

---

### 第三部：《看不见的城市》模式
**原著：** 伊塔洛·卡尔维诺（Italo Calvino）

**文学基因：** 马可·波罗向忽必烈汗描述55座城市，每座城市以一个女性名字命名，每一座都是对某种概念的冥想：记忆之城、欲望之城、符号之城、轻盈之城……所有的城市都是威尼斯，威尼斯是所有的城市。

**照片映射：**
你的照片库是一张你自己的城市地图。每个「城市」是一个视觉主题群——不是地理位置，而是影像质地：「窗之城」是你拍过的所有窗户；「孤独之城」是所有构图里只有一个主体的照片；「光之临界城」是所有接近夕阳或黎明的照片。每座城市有一段诗意的描述文字。

**技术基础（分阶段）：**

*第一阶段（EXIF可做）：*
- 时刻城市：黎明之城、深夜之城、正午之城
- 设备城市：索尼之城（冷光、精确）vs 富士之城（暖调、颗粒）
- 长度城市：广角之城（环境、孤独）vs 长焦之城（凝视、距离）

*第二阶段（需要视觉AI）：*
- 内容城市：窗之城、树之城、人群之城、食物之城
- 情绪城市：留白之城、拥挤之城、几何之城
- 色调城市：冷色之城、暖色之城、单色之城

**UI 方向：**
一张暗色「地图」，每座城市是发光的点。点击进入某座城市，先看到一段诗意的城市描述（可以由 Claude API 生成），再展开照片。城市之间有细线连接——相关联的城市彼此感应。

---

### 第四部：《山海经》模式
**原著：** 中国先秦地理志异

**文学基因：** 以地理为经，以神话为纬。东山经、西山经、北山经——每个地方都有自己的山川、异兽、矿物、神明。地理是叙事的骨架，神话是赋予它意义的血肉。

**照片映射：**
你去过的地方，构成你自己的「山海经」。每个 GPS 坐标群是一座「山」，每次到访是一次「经行」，你的照片是这座山的「记载」。「北山经 · 伦敦之山：多窗，多冬雨，异兽为单反相机，其色冷。」没有GPS数据的地方，则是「未测之地」，留白本身也是叙事。

**技术基础：**
- GPS 聚类：已有基础，`gps_city` + `gps_country` 字段
- 地理层级：国家 → 城市 → 经纬度半径聚类（Haversine 距离）
- 「神话生成」：用 Claude API 为每个地点生成一段《山海经》风格的文字
- 无GPS照片：归入「未测之地」，以《沙丘》模式处理

**UI 方向：**
一张个人世界地图（暗色、手绘感）。你到过的地方亮起，未到的地方是黑暗。点击某个地点，进入该地的「山志」，看到你在那里拍摄的所有照片，配以AI生成的神话描述。

**当前限制：** 我们目前 GPS 数据极少（46,000张照片中只有极少数有坐标），此模式需要等 GPS 数据积累或用户手动标记地点后才能完整体验。

---

### 第五部：《沙丘》模式（无名之境）
**文学基因参照：** 赫伯特《沙丘》及类似末日科幻小说的考古感——从废墟里挖出的银盐照片，来自一个失落的文明。你不知道它们是谁拍的，在哪里，为什么。神秘感本身就是叙事。

**照片映射：**
没有拍摄时间、没有GPS、没有相机信息——这些照片是真正的「无名文物」。不把缺失当缺陷，把它变成气氛。系统为这些照片分配一种「出土编号」，以考古报告的语气描述它们：「文物编号 #0047。介质：数字。时代：不明。出土地层：2023年外置硬盘第三区。图像内容无法识别来源。」

**技术基础（最容易实现）：**
- 查询：`WHERE shot_at IS NULL OR (gps_lat IS NULL AND camera_model IS NULL)`
- 神秘度评分：缺失字段越多，神秘度越高（0-100分）
- 出土编号生成：简单序列 + 哈希截断
- 考古描述：模版文字 或 Claude API 生成

**UI 方向：**
高对比度、颗粒感、几乎是黑白的界面。照片以「档案袋」形式呈现，边角磨损感。文字是等宽字体，像档案文件。背景有极轻的噪点纹理。这是五种模式里最「可以立刻做」的，因为只需要一个UI风格和一个查询。

## 入门体验：100张照片原则

**为什么100张？**
- 不需要等待4万张全部扫描完成
- 100张足以触发《哈扎尔词典》和《酒吧长谈》的基础叙事
- 让用户在等待5分钟内就能体验到「哦，还有这种方式」的感受
- 愿意继续的用户，才会投入导入更多照片

**100张的选取策略：**
不是随机100张，而是「最有叙事张力」的100张：
- 跨越尽可能多的年份
- 覆盖尽可能多的时刻分类（破晓/黄昏/夜晚）
- RAW 文件优先（更丰富的EXIF）
- 神秘度高的照片（缺失数据，直接触发《沙丘》模式）

## 新增所需数据结构

```sql
-- 照片视觉画像（从EXIF计算，一次性批量填充）
ALTER TABLE photos ADD COLUMN focal_category TEXT;     -- 'wide'|'normal'|'tele'
ALTER TABLE photos ADD COLUMN exposure_category TEXT;  -- 'daylight'|'low_light'|'night'
ALTER TABLE photos ADD COLUMN depth_category TEXT;     -- 'shallow'|'medium'|'deep'
ALTER TABLE photos ADD COLUMN moment_category TEXT;    -- 'dawn'|'morning'|'afternoon'|'golden'|'evening'|'night'
ALTER TABLE photos ADD COLUMN deliberate INTEGER;      -- 1=RAW, 0=JPEG-only
ALTER TABLE photos ADD COLUMN mystery_score REAL;      -- 0-1, 缺失EXIF越多越高

-- 时间回响关系（《酒吧长谈》）
CREATE TABLE temporal_echoes (
    photo_a TEXT REFERENCES photos(id),
    photo_b TEXT REFERENCES photos(id),
    echo_strength REAL,  -- 0-1, 共同维度越多越强
    shared_dims TEXT,    -- JSON: ['moment','focal','exposure']
    PRIMARY KEY (photo_a, photo_b)
);

-- 城市词条（《看不见的城市》，阶段二）
CREATE TABLE novel_cities (
    city_id TEXT PRIMARY KEY,
    name TEXT,           -- 诗意名称，如「黎明之城」
    description TEXT,    -- Claude API 生成的城市描述
    basis TEXT,          -- 'exif'|'vision_ai'|'gps'
    photo_count INTEGER
);
```
