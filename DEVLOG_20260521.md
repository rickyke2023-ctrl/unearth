  - **预加载机制**：提前缓存后两张照片，切换无卡顿

### 修复两个隐蔽 bug

**Bug 1 — 照片无遮盖直接裸显：**  
Canvas 条件渲染（`{imgLoaded && <canvas>}`）导致 ResizeObserver effect 触发时 canvas 还没挂载到 DOM。修复：把 `imgLoaded` 加进 effect 依赖，确保 canvas 存在后再初始化。

**Bug 2 — 第 10 张照片无法做决策：**  
预加载把图片放进浏览器缓存，React 合成 `onLoad` 对已缓存图片不可靠。`imgLoaded` 永远停在 `false`，按钮永不出现。修复：挂载后检查 `img.complete`，命中则直接设 `true`，绕过 `onLoad`。

这两个 bug 的共同特征：**在开发环境不复现，只在特定条件（条件渲染时序、预加载缓存）下触发**。靠人工反复测试发现的，这正是 `check_all.sh` 存在的理由。

### 带走的记忆（KeptView）

## 今天的问题

**两个 bug 都靠人工触发才发现。**  
Canvas 时序问题和缓存 onLoad 问题，`tsc` 和 `npm build` 都不会报错，只有实际操作才暴露。`check_all.sh` 现在能覆盖 API 存活，但无法覆盖「第 N 张图片的渲染时序」这类运行时行为。  
→ 后续要考虑 E2E 测试覆盖关键用户路径（Playwright），否则每次视觉变动都依赖人工回归。

**dispatch.sh 的交互式 read -p 无法自动化。**  
Claude 直接运行 dispatch.sh 会卡在等待用户输入，需要 Ricky 自己跑。这个设计是有意为之（防误触发），但目前还是在制造一次额外的人工切换。  
→ 可以改成 `--yes` flag 模式：正常运行显示预览 + 等待确认，加 flag 时跳过确认（只在 Claude 主控时使用）。

**Context 窗口在一个 session 内用尽。**  
今天全程在同一个 Claude 对话里工作，到会话末期 context 已满，触发了 session 压缩续接。估算今天的 token 消耗（注意：每轮对话都把完整历史重新发给模型，实际消耗远高于最终 context 大小）：

| 来源 | 估算 |
|---|---|
| 主对话（约 50 轮 × 平均 80k context/轮，input） | ~4M tokens |
| 主对话 output（代码 + 分析）| ~150k tokens |
| Codex Task D + E（代码生成 + 验收） | ~1.5-2M tokens |
| **合计** | **~6-10M tokens** |

今天 context 满了之后触发了系统自动续接，损失了部分中间细节。

**正确做法是主动 compact，不要等系统被动压缩：**
- 时机：每个功能 commit + 验证完成之后，切换任务之前
- 好处：可以控制摘要内容，把最重要的设计决策显式写进去；系统自动压缩不知道什么该保留
- 下次开工前：用 `/compact` 整理，或直接开新会话，把 PROJECT_MEMORY.md 重新注入

## 今天的认识提升

**1. 协作模式不是工具问题，是架构问题。**  
昨天写了 STATUS.md 和 coordinator.py，以为「加了文档就减少了摩擦」。今天才真正跑通：Claude 作为主控，Codex 作为专项执行，CODEX_DISPATCH.md 是合同，dispatch.sh 是签字——每个角色职责清晰，人只在「要不要做这件事」的时刻出现，不在「怎么做」的执行链里。这个模式值得固化。

**2. Bug 的真实成本不是修复时间，是发现时间。**  
Canvas 时序 bug 和 onLoad 缓存 bug，两个加起来修复不超过 10 行代码，但发现它们花了 70 分钟。因为它们只在特定条件下触发（条件渲染 + 预加载缓存），开发环境完全正常。这类 bug 靠 lint 和 tsc 发现不了，靠 E2E 才能覆盖。`check_all.sh` 是好的开始，但还差一层运行时验证。

**3. 产品方向需要定期「停下来对齐」，不能只靠执行惯性。**  
今天在开发中途花了半小时重排路线图，结论是「不要在错误的交互模式下扫 4 万张照片」。这个决定如果晚做，成本是几十小时的无效规模扩展。**产品讨论的性价比在这种快速迭代里极高，不应该被执行冲动挤掉。**
