## 今天做了什么

### 叙事引擎骨架上线（最重要的事）

整个 app 从「照片整理工具」变成了另一种东西。三个新组件全部完成，TypeScript 零错误，一次性通过。

**GatewayView — 微光之门**  
入口不再是列表页，而是一个需要你主动触碰的光点。三阶动画：
- `dormant`：漆黑背景中，一个 7px 的琥珀色光芯在呼吸。2 秒后「触」字淡出——邀请，不催促
- `awakening`：你点下去，光芒外散（0.65s bloom），光芯爆裂消失
- `revealed`：从中心展开两个选择——「整理」（冷色，存档世界）和「发现」（暖色，叙事世界）

这不是 UI 设计，是哲学立场：**用户必须主动进入记忆，记忆不会自己涌向你。**

**LibraryView — 书库**  
五本书的书脊。现在只有《无名之境》亮着（琥珀色脊，可点开）。其余四本是封印的影子——灰暗，各带一行解锁条件的描述。这种封印感是有意为之的，它在说：「叙事的可能性是存在的，但你还没有做好准备。」

**DuneView — 无名之境**  
照片变成考古碎片。左侧缩略图格（2列×N行，棕褐色滤镜 + 胶片颗粒），右侧档案面板。每张碎片有：
- FRAGMENT 编号（如 UNK-0423）
- ERA（以你最早的照片年份为起点的纪年：「First Year」「Fifth Year」）
- TERRITORY（全部 0% GPS，所以全是「origin unknown」——这反而成了美学）
- INSTRUMENT（相机型号，没有则「unspecified device」）
- CONDITION（由 mystery score 决定：severely deteriorated / origin unknown / partially legible / well-preserved）
- MYSTERY LEVEL（进度条，缺 EXIF 字段越多越高）
- 键盘导航：← → 切换碎片，ESC 返回书库

后端 `novel.py` 的 `_mystery_score` 算法：camera_model 缺失 +0.40，GPS 缺失 +0.35，gps_city 缺失 +0.25。当前 46K 张照片中，GPS 覆盖率约 0%——所以几乎所有照片的 mystery score 都极高。**这不是数据缺失，这是叙事的原材料。**

## 真正的形式与叙事创新在哪里

这是今天最值得写的部分。我想诚实地评估：我们做的事情，在照片软件领域里有多少是真正的创新，还是只是换了个更好看的壳。

### 1. 缺失即叙事（The Absence as Narrative）

这是今天最反常识的设计决策。

所有照片软件对缺失元数据的态度是：**这是错误，要修复。**  
地图里没有 GPS 点，EXIF 没有焦距，相机型号是空的——这些都是「待填写的空格」。

《沙丘》的 mystery score 算法把这件事反转了：**元数据越缺失，照片越神秘，越适合作为考古碎片。** 没有相机型号的照片不是「信息不完整」，而是「来自不明文明的遗物，使用仪器不详」。没有 GPS 的照片不是「位置丢失」，而是「领地：来源不明」。

这不是文案包装，是一种关于记忆本身的立场：**我们自以为拥有的记忆，其实大部分是缺损的。** 你知道那张照片是2021年拍的，但你不记得那天发生了什么，去的哪里，用的什么相机。这种缺损不应该被修复，而应该被承认，甚至被放大。

**实现上的关键**：mystery score 不是装饰，是排序算法的权重。缺失最多的照片最先出现在碎片列表里。

---

### 2. 文学框架替代元数据框架（Literary Frames vs Metadata Frames）

这是产品层面最大的创新，但也是最难实现的。

普通照片软件的组织逻辑：**时间 → 地点 → 相机 → 人物**。这是元数据框架。每一张照片是一条记录，字段是坐标。

五部小说的逻辑是：**同一批照片，进入不同的文学意识框架，产生不同的情感真相。**

| 框架 | 什么变了 | 感受到的是什么 |
|---|---|---|
| 《沙丘》| 照片是碎片，用户是考古学家 | 自己的记忆是「失落文明」—— 陌生、神秘、距离感 |
| 《哈扎尔词典》| 照片是词条，维度是焦距/时刻/季节 | 自己有某种稳定的「视觉 DNA」—— 你总在傍晚拍照 |
| 《酒吧长谈》| 照片是时间回响，同时刻跨年份对话 | 不同的「你」在同一个时刻相遇 |
| 《看不见的城市》| 照片是城市，AI 命名主题 | 你一直在记录一些你自己都没有命名的地方 |
| 《山海经》| 照片是神话地理，GPS 是经纬 | 你走过的地方构成你的个人神话 |

这五种框架**不改变数据，只改变凝视数据的角度**。这在软件里是罕见的，因为大多数软件假设数据只有一种正确的解读方式。

**实现上的挑战**：五种框架对数据的要求不同。《沙丘》只需要 status=active + mystery score，今天已经上线。《哈扎尔词典》需要 EXIF 维度（当前扫描器未存焦距/光圈）。《看不见的城市》需要视觉 AI。《山海经》需要 GPS。这是一个渐进解锁的产品路线图，不是一次性完成的功能列表。

### 3. 邀请感的交互哲学

GatewayView 里那个光点，等你去触碰，不会自己移动。

这是一个有意识的摩擦设计。大多数 APP 的入口是：**内容迎面扑来，瀑布流、推送、算法推荐**——内容向用户移动，用户是被动的。

微光之门的立场相反：**记忆在那里，但你要先踏进去。** 这种门槛感是刻意的，它在训练用户一种心理准备：「我现在要做的事，是进入记忆，不是浏览内容。」

这不是新发明，但在照片软件里是新的。最接近的先例是一些博物馆 app 的入口设计——你进博物馆前有门，有安静的前厅，然后才是展厅。

### 4. 100张照片先行（尚未实现，但值得记录）

今天想清楚了一件事：这个 app 的核心价值主张（把记忆变成叙事）**不需要 46000 张照片才能体验到**，100 张就够了。但现在的 onboarding 要求用户先等待整个图库扫描，这是把「门槛」放在了错误的地方。

正确的顺序是：**先让用户在 10 分钟内被叙事打动，再让她决定是否投入整个图库。**

这不只是 UX 改进，是产品能否获得第一批真实用户的关键。明确为未来功能，暂不实现。

## EXIF 现状（哈扎尔词典的底线）

今天做了一次数据库摸底，结果：

| 字段 | 覆盖率 | 
|---|---|
| `shot_at` | 100% |
| `file_type` | 100% |
| `camera_model` | 74% |
| `gps_lat` | ~0% |
| `focal_length_mm` / `aperture` / `iso` | ❌ 未存储 |

焦距、光圈完全未入库。但 `shot_at` 100% 覆盖意味着我们有完整的 **时刻维度**——7年里每一张照片的拍摄时间。

哈扎尔词典 MVP 可以用：
- **时刻词条**：清晨 / 上午 / 午后 / 傍晚 / 夜晚
- **机身词条**：你用过的每台相机
- **介质词条**：RAW_SONY / RAW_FUJI / JPEG / HEIF
- **季节词条**：春夏秋冬

「所有傍晚五点到七点拍的照片，2019—2026」本身就是一本词条。等之后扫描器补全 EXIF，词典会更丰富。
