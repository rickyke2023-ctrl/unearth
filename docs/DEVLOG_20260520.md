## 今天的亮点

**产品定义做得好。**  
从「照片整理工具」到「记忆考古」的概念跳跃，是今天最重要的事。这个定义让后续所有设计决策有了方向——不是在优化 Lightroom，是在做一个完全不同的东西。

**文档驱动开发。**  
SPEC.md → API.md → PROJECT_MEMORY.md → DESIGN_LANGUAGE.md，四份文档在代码之前写好，两个 AI 工具有了共同语言，减少了很多来回返工。

**压力测试做了。**  
没有直接上全盘，先跑 50 张，再跑 2023 全年 8105 张。发现了 649 张年份错误的照片，这在全盘扫描时会是更大的问题。小步走救了一次。

**安全意识在线。**  
软删除机制（staging 文件夹）、禁止直接删除原文件、audit log——这些在第一天就落地，没有为了图方便跳过。

**V2 产品方向想清楚了。**  
主题模式（你的执念）+ 故事模式（跨时期叙事）+ 每日限额——这个方向比「Lightroom 换壳」高出一个维度，而且已经写进 SPEC.md。

## 今天的问题

**人肉 API 问题没有根本解决。**  
STATUS.md + coordinator.py 只是减少了一个步骤，你还是需要在三个窗口之间切换传递信息。真正的解决方案是 MCP 接入，但今天没时间做。

**UI 和 Lightroom 的距离还不够远。**  
SiteView 和 DecisionView 改了，但核心交互逻辑（一张一张翻）还是线性的。「从岩层里取出」的感觉还没有实现——那需要一次更大的交互重设计。

**设计语言执行不够严格。**  
DESIGN_LANGUAGE.md 写了，但 Claude Code 有时候会按自己的理解实现，而不是严格按规范。需要在每次指令里明确引用规范，而不是假设她会自己读。

**额度消耗比预期快。**  
Claude Pro 和 Codex 大部分额度在一天内用完。截图功能尝试了 html2canvas、screencapture、Playwright 三种方式才成功，这类重复试错消耗了不少 token。

**卡点复盘：**

| 卡点 | 耗时估算 | 原因 | 下次改进 |
|---|---|---|---|
| Claude Desktop binary 路径 | ~15分钟 | nvm 安装路径不在系统 PATH | 启动前检查 symlink |
| 扫描「卡住」误判 | ~20分钟 | Claude Code 超时，以为 hang 了 | 改用轮询替代 SSE 监听 |
| 幽灵文件发现修复 | ~25分钟 | 数据一致性 bug，值得花时间 | ✅ 正确处理 |
| 截图工具试错 | ~20分钟 | 三种方案轮流失败 | 固定用 Playwright，写进脚本 |
| 三窗口中继等待 | ~2小时 | 架构问题，无法绕过 | MCP 接入后根本解决 |
